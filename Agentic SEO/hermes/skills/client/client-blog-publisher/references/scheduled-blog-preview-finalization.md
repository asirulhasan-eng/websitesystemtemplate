# Scheduled blog preview finalization

Use after a scheduled `new_blog_post` pickup creates a semi-safe preview branch/PR and updates task state.

## Durable finalization pattern

1. Verify task/deployment/outbox state from the authoritative DB (`/opt/client-sqlite/seo-agent.db`) after the preview branch is pushed:
   - task row should be `preview_ready`
   - deployment row should include branch, commit, preview URL if known, and `validation_status`
   - task/deployment outbox jobs should be completed or explicitly reported as blocked
2. Run a final validation pass on the preview branch, but leave the site repo on `master` afterward so the next scheduled run starts clean:
   ```bash
   cd /opt/client-site
   git checkout agent/<task_id>-<slug>
   # run HTML/JSON-LD/FAQ/index/sitemap/floating-CTA checks and npm test when available
   git checkout master
   git pull --ff-only origin master
   ```
3. If outbox processing or validation metadata changed the authoritative DB after the first commit, commit and push `/opt/client-sqlite/seo-agent.db` again. Do not leave the DB repo dirty just because the preview already exists.
4. Commit/push Obsidian mirror notes generated by outbox processing if present.
5. Final repo hygiene check should include all four repos and should normally end clean/synced:
   - `/opt/client-site` on `master...origin/master`
   - `/opt/client-agent` on `main...origin/main`
   - `/opt/client-sqlite` on `main...origin/main`
   - `/opt/client-obsidian` on `main...origin/main`

## Reporting notes

Report the PR/preview branch as semi-safe and unmerged. Include task ID, branch, commit, PR/preview URL, files changed, validation commands/results, durable DB/Obsidian commit evidence, and explicitly state that production was untouched.
